# 代码规范与 Review 基线

这份规范的目标很简单：让开发、review、自动化 agent 都按同一把尺子做事，少打口水仗，多做可验证的改动。

## 1. 总原则

1. **小步快跑**：一次 PR 只解决一个明确问题，避免大杂烩。
2. **最小必要改动**：修 bug 就修 bug，做功能就做功能，不顺手大重构。
3. **可验证**：每次改动都要说明如何验证，至少给出本地运行、接口验证或截图/日志中的一种。
4. **结构清晰**：业务逻辑、数据模型、路由/UI 展示尽量分层，不把所有逻辑塞进一个文件。
5. **Review 评事实，不评口味**：优先看正确性、可维护性、风险和验证，不纠缠纯个人审美。

## 2. Python 后端规范

### 2.1 工具

- formatter: `black`
- import 排序: `isort`
- lint: `ruff`
- 行宽：100
- Python 目标版本：3.12

### 2.2 命名

- 变量 / 函数：`snake_case`
- 类 / Pydantic Model：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 文件名：优先 `snake_case.py`

### 2.3 结构约定

- `app/main.py`：应用入口、生命周期、路由挂载
- `app/models/`：数据库模型、schema、数据库连接
- `app/services/`：业务逻辑与外部数据处理
- 避免把复杂逻辑直接堆进路由函数
- 单个函数过长时优先拆成小函数，不要让一个函数同时做“查询 + 计算 + 格式化 + 落库”

### 2.4 Review 关注点

重点看：
- 有没有把副作用藏得太深
- 数据库操作是否足够明确
- 初始化/启动逻辑是否幂等
- 错误处理是否至少做到“失败可见”

## 3. 前端规范

### 3.1 工具

- formatter: `prettier`
- lint: `next lint`
- TypeScript 报错必须处理，不允许靠 `any` 一把梭糊过去，除非有明确理由

### 3.2 结构约定

- 页面层负责组合，不负责承载太多业务细节
- 复用逻辑优先抽到组件或工具函数
- 样式写法尽量统一，不要同时混用多套完全不同的组织方式
- 新增 UI 时优先保证信息层级和可读性，再考虑花活动画

### 3.3 Review 关注点

重点看：
- 组件职责是否单一
- 数据展示是否稳定、空态是否可读
- 是否存在明显写死数据却没有注释说明
- 样式改动是否影响已有布局

## 4. Git / Commit / PR 规范

### 4.1 分支命名

统一使用：
- `feat/...`
- `fix/...`
- `docs/...`
- `chore/...`
- `refactor/...`

### 4.2 Commit 规范

使用 Conventional Commits：
- `feat:` 新功能
- `fix:` 修复
- `docs:` 文档
- `chore:` 杂项维护
- `refactor:` 重构（不改变外部行为）

### 4.3 PR 规范

每个 PR 至少写清楚：
- 做了什么
- 为什么做
- 怎么验证
- 风险或后续事项

没有验证说明的 PR，默认不算完整。

## 5. Review 规则

Review 时按下面顺序判断：

1. **需求正确吗**：改动是不是解决了目标问题
2. **实现清晰吗**：结构是否容易理解和维护
3. **风险可控吗**：是否引入额外副作用或隐藏耦合
4. **验证充分吗**：有没有最基本的运行 / lint / smoke / 手测依据
5. **范围克制吗**：有没有顺手塞入无关改动

以下情况可以直接要求修改：
- 改动范围明显失控
- 没有任何验证
- 引入魔法常量但无解释
- 把多层职责硬塞进单个函数/组件
- 代码能跑，但后续维护成本明显炸裂

## 6. 自动化 Agent 约定

无论是人工开发还是自动推进 agent，都遵守：

- 先读目标，再改代码
- 只做本轮明确边界内的事
- 提交前必须做至少一种验证
- PR 描述必须带验证结果
- reviewer 以本规范为基线，不按个人心情飘

## 7. 当前建议执行命令

后端：

```bash
cd backend
uv pip install --python .venv/bin/python black isort ruff
black app
isort app
ruff check app
```

前端：

```bash
cd frontend
npm run lint
```

如果后续要上 CI，就直接把这些检查接进去。
